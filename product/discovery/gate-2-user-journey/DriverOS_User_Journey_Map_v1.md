## 1. Job-to-be-Done (แก้ไขจากสมมติฐานเดิม)

> ⚠️ **อัปเดตล่าสุด:** JTBD ด้านล่างนี้ถูกขยายเพิ่มเติมใน `product/PRODUCT_EVOLUTION.md` — จาก "รู้กำไรสุทธิ" เป็น "ช่วยให้คนขับตัดสินใจได้ดีขึ้นทุกวัน" (Net Profit กลายเป็น Metric ไม่ใช่ Mission) เก็บเนื้อหาเดิมด้านล่างไว้เพื่อ traceability

> ❌ เดิมคิดว่า: "ช่วยคนขับบันทึกรายได้แต่ละทริป"
> ✅ จริง: **"ช่วยคนขับรู้กำไรสุทธิ (หลังหักค่าน้ำมัน/ค่าเสื่อม) โดยไม่ต้องคำนวณเอง"**

ยอด gross รายได้ คนขับรู้จาก Grab อยู่แล้ว — ไม่ใช่ปัญหา
ปัญหาจริงคือ **ไม่มีใครคำนวณต้นทุนให้ ทำให้ไม่รู้ว่าขับแล้วคุ้มแค่ไหน**

**หมายเหตุเชิงกลยุทธ์:** การเก็บข้อมูลระดับ trip (trip-level data) ยังจำเป็น แต่เป็น **infrastructure สำหรับ driving intelligence platform ในอนาคต** ไม่ใช่ hero feature ของ UX วันนี้ ต้องออกแบบให้อยู่ "เบื้องหลัง" ไม่ใช่จุดศูนย์กลางของหน้าจอ

## 2. As-Is Journey (สถานการณ์จริงวันนี้ ไม่มี DriverOS)

| ช่วง | สิ่งที่คนขับทำ | เครื่องมือปัจจุบัน | Pain Point | อารมณ์ |
|---|---|---|---|---|
| **ก่อนออกกะ** | เติมน้ำมัน, เปิดแอป Grab | Grab app | ไม่รู้ต้นทุนตั้งต้นชัดเจน | เฉยๆ |
| **ระหว่างขับ** | โฟกัสรับงาน | Grab app | ไม่มี (ไม่ใช่จุดที่ต้องแทรกแซง) | โฟกัส |
| **ช่วงว่าง/รองาน** | **หาจุด/พื้นที่ที่มีงานเยอะเป็นหลัก** — เช็ครายได้เป็นรอง (ทุก 2-3 เที่ยว เฉพาะตอนมีช่วงว่างจริงๆ) | ความจำ + Grab app | ไม่รู้กำไรสุทธิ (ยังไม่หักต้นทุน) | อยากรู้แต่ไม่ใช่ priority อันดับ 1 |
| **เติมน้ำมันระหว่างกะ** | แวะเติมแบบไม่มีรอบตายตัว — แล้วแต่สถานการณ์วันนั้น | จ่ายเงินสด/ตัดบัตร | ไม่ได้บันทึกผลกระทบต่อกำไรทันที | ไม่รู้สึกอะไรเป็นพิเศษ |
| **จบกะ** | รู้แค่ยอดรวมที่ได้ (gross) **ไม่ได้คำนวณกำไรสุทธิเลย** | ไม่มีเครื่องมือ | อยากรู้กำไรจริง แต่ขี้เกียจคำนวณเอง | Latent demand ที่ไม่ถูกเติมเต็ม |

### Insight สำคัญที่สุด: การแย่งชิงความสนใจ (Attention Conflict)
ช่วงเวลาว่างที่มีค่าที่สุดของคนขับ ถูกจองไว้ให้ **"หางาน" ไม่ใช่ "เช็คกำไร"** ดังนั้นแอปใดก็ตามที่บังคับให้คนขับ "เปิดแอปแยก" เพื่อดูข้อมูลการเงิน จะแพ้การแข่งขันด้าน attention กับ Grab เองเสมอ

## 3. สมมติฐานเดิมที่ถูก Validate/Invalidate

| สมมติฐานจากบรีฟเดิม | ผล | เหตุผล |
|---|---|---|
| Import Trip ทีละทริปด้วย OCR/Screenshot | ❌ Invalidated (เป็น hero feature) | คนขับไม่เช็คทุกทริป และ gross ไม่ใช่ปัญหา — แต่ยังมีค่าในฐานะ data infrastructure |
| เลือก Driving Style ตอนเริ่ม session | ❌ Invalidated | พฤติกรรมปรับเปลี่ยนไปเรื่อยๆ ระหว่างกะ ไม่ใช่การตัดสินใจล่วงหน้าแบบ fixed |
| Session แบบ Start/End กดเอง | ⚠️ ต้องพิจารณาใหม่ | ยังจำเป็นเป็น container แต่ไม่ควรเป็นจุดปฏิสัมพันธ์หลัก |
| Fuel/Expense มี location trigger อัตโนมัติ | ⚠️ ปรับ | จังหวะเติมน้ำมันไม่แน่นอน ต้องเป็น "log ได้ทันทีทุกเมื่อ" ไม่ใช่ตามรอบ |
| Dashboard สรุปท้ายกะ | ✅ Validated | ตรงกับ latent demand "อยากรู้กำไรสุทธิ แต่ขี้เกียจคำนวณ" |
| แอปเดี่ยว ไม่ใช่ multi-app aggregator (ในเฟสนี้) | ✅ Validated | คนขับใช้แอปเดียว ลดความซับซ้อนได้จริง |

## 4. การตัดสินใจสถาปัตยกรรมหลัก (Critical Decision)

**DriverOS ควรเป็น Notification / Lock Screen–first experience** ไม่ใช่แอปแยกที่ต้องเปิดเอง

**เหตุผล:** เพื่อไม่แข่งกับ Grab ในช่วงเวลาว่างที่คนขับใช้หา hotspot งาน — ข้อมูลกำไรสุทธิต้อง "โผล่มาเอง" แบบ passive ผ่านการแจ้งเตือน ไม่ใช่สิ่งที่ต้องเปิดแอปไปตามหา

**ผลต่อโครงสร้าง UI:**
- แอปแบบเต็ม (full app) จะถูกเปิดจริงจังแค่ 2 momen: **(1) ตั้งค่าเริ่มกะ** และ **(2) ดูสรุปฉลองท้ายกะ**
- Notification เป็น **จุดปฏิสัมพันธ์หลัก** ตลอดกะ พร้อม quick action (เช่น "+ บันทึกน้ำมัน") ในตัวแจ้งเตือนเลย โดยไม่ต้องปลดล็อกเข้าแอป

## 6. ประเด็นเปิดที่ต้องตรวจสอบก่อน Gate 3 (Wireframe)

1. **กลไกรับข้อมูล gross earning โดยไม่รบกวนคนขับ** — แนวคิดที่ต้อง validate feasibility ภายหลัง (ไม่ใช่ตอนนี้): การอ่านแจ้งเตือนจบทริปจาก Grab เอง (background) แทนการให้คนขับ screenshot เอง
2. **Interaction design ของ quick-log ค่าน้ำมัน** ผ่าน notification ต้องเร็วแค่ไหนถึงจะถูกใช้จริงระหว่างขับ
3. **Trigger การจบกะ** — auto-detect (เช่น หยุดเคลื่อนที่นาน) หรือคนขับกดเอง

## 7. สิ่งที่ตัดออกจากขอบเขตตอนนี้อย่างชัดเจน

- Multi-app aggregation (Grab+Bolt+LINE MAN+inDrive พร้อมกัน)
- ปุ่มเลือก Driving Style แบบ fixed ตอนเริ่มกะ
- Per-trip OCR ในฐานะ primary UI (ยังมีในฐานะ infrastructure เบื้องหลัง)
